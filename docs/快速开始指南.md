# 快速开始指南

基于 FastAPI + Vue3 的 Linux 运行监控系统，支持**Windows + WSL 双环境**部署。

---

## 架构说明

本系统支持两种部署模式：

| 模式 | 说明 |
|------|------|
| **原生模式** | 直接在 Windows/Linux 上运行，数据采集使用本地 psutil |
| **WSL + Agent 模式** | WSL 跑后端，Windows 跑 Agent，适用于 WSL 里监控 Windows 宿主机真实硬件 |

---

### 为什么 WSL 不能直接获取 Windows 宿主机硬件数据

WSL（Windows Subsystem for Linux）是 Windows 内置的一个轻量级虚拟机，运行一个真实的 Linux 内核。当在 WSL 里执行 `psutil` 库时，它底层调用的是 Linux 系统接口（`/proc` 文件系统、`/sys` 等），因此**只能读到 WSL 虚拟机的数据**，而非 Windows 宿主机真实硬件。

举例说明：

| 监控指标 | 直接在 WSL 里用 psutil 读到的是 |
|----------|----------------------------------|
| CPU 使用率 | WSL 虚拟 CPU 的占用，不是 Windows 整体 CPU |
| 内存总量 | WSL 分配到的虚拟内存，不是整台机器的物理内存 |
| 磁盘 I/O | WSL 虚拟磁盘的读写，不是宿主机真实磁盘 |
| 网络流量 | WSL 虚拟网卡，不是 Windows 真实网卡 |
| 进程列表 | WSL 内部的进程，不是 Windows 上的进程 |

这是 Linux 与 Windows 之间的系统边界问题，psutil 作为跨平台库，设计上遵循"在哪台机器运行就读哪台机器"的原则，无法穿透虚拟机直接读取宿主机。

---

### 解决方案设计思路

要实现"在 WSL 里监控 Windows 真实硬件"，核心思路是：**把采集端和数据源分开**。

- **Windows 侧**：运行一个轻量的 HTTP 服务（Agent），直接调用 Windows 上的 psutil，读取真实的硬件数据，通过 HTTP 接口暴露出来
- **WSL 侧**：不再自己调 psutil，而是通过 HTTP 请求调用 Windows Agent，将拿到的数据当作自己的采集结果使用

这样 WSL 侧仍然是一个标准的 Linux Python 后端，使用相同的接口（`get_cpu()`、`get_memory()` 等），只是底层数据来源从"本地 psutil"变成了"Windows Agent 的 HTTP 接口"。

**WSL + Agent 架构图：**

```
WSL (FastAPI 后端)                    Windows (Agent)
┌─────────────────────┐              ┌──────────────────┐
│ wsl_windows_client  │ ── HTTP ──>  │ windows_agent.py  │
│ (采集代理层)         │  172.x.x.x   │  用 psutil 读真实 │
│  复用原有接口        │   :8001      │  Windows 硬件数据  │
│  切换数据来源        │              │                   │
└─────────────────────┘              └──────────────────┘
```

关键设计点：

1. **自动检测**：WSL 侧通过 `ip route show` 检测 Windows 宿主机 IP，无需手动配置
2. **接口一致**：Windows Agent 复刻与 collector.py 相同的接口定义，WSL 侧可以无缝切换
3. **自动降级**：若 Windows Agent 不可用，WSL 侧自动降级到本地 psutil，保证服务可用性
4. **零侵入原有代码**：collector.py 保持不动，新增 wsl_windows_client.py 作为代理层

## 环境准备

### 方式一：原生模式（Linux / Windows 直接部署）

#### 后端

```bash
# 进入后端目录
cd backend

# 创建虚拟环境
python3 -m venv .venv

# 激活虚拟环境
# Linux/macOS:
source .venv/bin/activate
# Windows:
.venv\Scripts\activate

# 安装依赖
pip install -r requirements.txt

# 启动服务
uvicorn main:app --host 0.0.0.0 --port 8000
```

#### 前端

```bash
# 进入前端目录
cd frontend

# 安装依赖
npm install

# 启动开发服务器
npm run dev

# 构建生产版本
npm run build
```

---

### 方式二：WSL + Agent 模式（监控 Windows 宿主机）

#### 第一步：Windows 侧启动 Agent

```powershell
# PowerShell 管理员窗口
cd D:\AI\job\linux-monitor-delivery\backend
python windows_agent.py
```

> Agent 监听 `http://0.0.0.0:8001`，无需修改 IP，WSL 会自动检测 Windows 宿主机网关地址。

#### 第二步：WSL 侧启动后端

```bash
cd backend
USE_WINDOWS_AGENT=true uvicorn main:app --host 0.0.0.0 --port 8000
```

> `USE_WINDOWS_AGENT=true` 环境变量使后端自动切换到 Agent 模式。若 Agent 不可用，会自动降级到 WSL 本地 psutil。

#### 第三步：WSL 侧启动前端
```bash
npm run dev
```

## API 接口示例

### 获取当前指标

```bash
curl http://localhost:8000/metrics/current
```

### 获取历史数据

```bash
# 注意：时间戳单位为毫秒
curl "http://localhost:8000/metrics/history?metric_type=cpu&start_time=$(date -d '1 hour ago' +%s)000&end_time=$(date +%s)000"
```

### 获取进程列表

```bash
curl "http://localhost:8000/processes?top=10&sort=cpu"
```

## 常见问题

### 1. 前端无法连接后端

检查 WSL 和 Windows 之间的网络连通性：
```powershell
Test-NetConnection -ComputerName 172.21.144.1 -Port 8001
```

### 2. WSL + Agent 模式下数据仍取到 WSL 虚拟机数据

确认 Agent 已启动且 WSL 侧 FastAPI 带有 `USE_WINDOWS_AGENT=true` 环境变量。

### 3. Agent 端口被防火墙拦截

```powershell
netsh advfirewall firewall add rule name="Agent8001" dir=in action=allow protocol=tcp localport=8001
```

### 4. 数据未更新

刷新页面，检查 WebSocket 连接状态。
