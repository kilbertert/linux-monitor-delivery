# 基于 FastAPI + Vue3 的 Linux 运行监控系统的设计与实现

## 毕业设计（论文）



---

## 摘要

随着云计算和容器化技术的广泛应用，企业的服务器大多运行在 Linux 平台之上，及时掌握系统的资源使用状况、发现潜在瓶颈并采取预防措施，成为保障业务稳定运行的重要前提。本文设计并实现了一套基于 Python 后端框架 FastAPI 和前端框架 Vue3 的轻量级 Linux 运行监控系统。该系统能够实时采集 CPU、内存、磁盘 I/O、网络带宽等系统指标，通过 WebSocket 实现数据的实时推送，并使用 ECharts 进行可视化展示。系统采用 SQLite3 作为后端存储，支持历史数据查询和数据导出功能，同时实现了告警机制和进程监控功能。经过测试，系统运行稳定，各项功能满足设计要求。

**关键词**：Linux 监控；FastAPI；Vue3；psutil；ECharts



---

## ABSTRACT

With the widespread application of cloud computing and containerization technology, most enterprise servers run on Linux platforms. Timely grasping the system resource usage, discovering potential bottlenecks, and taking preventive measures have become important prerequisites for ensuring stable business operations. This paper designs and implements a lightweight Linux operating monitoring system based on Python backend framework FastAPI and frontend framework Vue3. The system can collect real-time CPU, memory, disk I/O, network bandwidth and other system metrics, push data in real-time through WebSocket, and visualize with ECharts. The system uses SQLite3 as backend storage, supports historical data query and data export, and implements alarm mechanism and process monitoring. After testing, the system runs stably and meets all design requirements.

**Keywords**: Linux Monitoring; FastAPI; Vue3; psutil; ECharts

## 第一章 绪论

### 1.1 研究背景与意义

随着互联网技术的飞速发展，企业级应用和云计算环境的规模不断扩大，服务器作为信息基础设施的核心载体，其稳定性和性能直接影响着整个业务系统的可用性。在这样的背景下，对服务器运行状态进行实时监控的需求变得日益迫切。Linux作为全球使用最广泛的服务器操作系统之一，其系统资源的监控与管理成为运维工程师和系统管理员日常工作中不可或缺的重要内容。

传统的Linux系统监控主要依赖于命令行工具，如top、htop、vmstat、iostat等，这些工具虽然功能强大，但存在明显的局限性。首先，它们只能提供实时的瞬时数据，无法对历史数据进行存储和分析；其次，多个监控指标分散在不同的命令中，缺乏统一的展示界面，需要运维人员手动整合信息；再者，这些工具不具备告警功能，无法在系统资源出现异常时及时通知相关人员。因此，设计并实现一套功能完善、操作便捷的Linux监控系统具有重要的实际应用价值。

本研究课题设计的Linux监控系统采用现代化的前后端分离架构，后端基于Python语言和FastAPI框架构建RESTful API服务，同时支持WebSocket协议实现数据的实时推送；前端采用Vue 3框架和ECharts可视化库，为用户提供直观友好的图形化操作界面。该系统能够对服务器的CPU、内存、磁盘I/O、网络带宽等关键资源指标进行实时监控，并以图表形式动态展示各项数据的变化趋势。同时，系统还具备历史数据存储、告警阈值设置、进程信息查看以及数据导出等实用功能，能够有效辅助运维人员掌握服务器运行状态，及时发现并处理潜在问题。

### 1.2 国内外研究现状

在服务器监控领域，国内外已经涌现出众多成熟的商业产品和开源解决方案。商业产品方面，SolarWinds Server & Application Monitor、SolarWinds Network Performance Monitor、Datadog、New Relic等国际知名软件提供了功能全面的监控服务，能够支持大规模分布式架构的监控需求，并具备智能告警和自动化响应能力。然而，这些商业软件通常价格昂贵，且部署和配置过程复杂，对于中小型企业和个人用户而言存在较高的使用门槛。

开源社区同样贡献了许多优秀的监控工具。Prometheus与Grafana的组合是目前最流行的开源监控解决方案之一，Prometheus负责从目标系统采集指标数据并存储时序数据库，Grafana则提供强大的数据可视化能力，二者通过松耦合的方式集成，能够满足多样化的监控场景需求。Zabbix作为企业级开源监控平台，具备分布式监控、自动发现、告警配置等丰富功能，被广泛应用于各种规模的IT基础设施中。Nagios则是经典的监控框架，以其灵活的插件机制著称，允许用户自定义监控脚本和告警规则。

然而，现有的监控方案各有优缺点。商业软件功能强大但成本高昂；Prometheus+Grafana虽然开源免费，但技术栈较重，部署和维护需要较高的技术门槛；Zabbix功能全面但界面相对过时，配置复杂度较高。针对毕业设计这一特定场景，我们需要一个技术栈适中、易于理解和实现、同时又具备实用价值的监控系统的实现方案。本设计选择自行开发一套轻量级的Linux监控系统，既能够在技术深度上满足毕业设计的学术要求，又能够为实际应用提供参考。

### 1.3 研究目标与内容

本研究的主要目标是设计并实现一个基于Web的Linux服务器监控系统，该系统应具备以下核心功能：

第一，实时监控功能。系统需要能够实时采集服务器的CPU使用率（包括总体使用率和各核心使用率）、内存使用情况（包括已用内存、可用内存、使用率等）、磁盘I/O统计（包括读写速率、IOPS等）以及网络带宽监控（包括上行带宽、下行带宽等），并将采集到的数据实时更新到前端界面。

第二，数据存储功能。系统采用SQLite作为后端数据库，能够将监控数据持久化存储，支持历史数据的查询和分析。用户可以查看特定时间段内的监控数据回溯，了解系统资源的长期变化趋势。

第三，可视化展示功能。系统前端使用ECharts图表库，将各类监控指标以折线图、饼图、仪表盘等形式直观展示，支持时间范围选择和平滑过渡动画，提升用户体验。

第四，告警机制功能。用户可以为各监控指标设置告警阈值，当实时数据超过阈值时，系统能够自动触发告警，并通过界面提示的方式通知用户。告警历史也会被记录到数据库中供查询。

第五，进程监控功能。系统能够列出当前系统运行的进程信息，支持按照CPU使用率或内存使用量排序，方便用户识别资源消耗大户。

第六，数据导出功能。系统支持将历史监控数据导出为CSV格式文件，便于用户进行离线分析和二次处理。

## 第二章 相关技术概述

### 2.1 Python 语言与 FastAPI 框架

Python是一种高级解释型编程语言，由Guido van Rossum于1991年首次发布。Python具有语法简洁、代码可读性高、生态丰富等优点，被广泛应用于Web开发、数据分析、人工智能、自动化运维等领域。Python 3.10版本引入了更强大的模式匹配功能和性能优化，使得Python在处理高并发场景时具有更出色的表现。

FastAPI是一个现代、高速的Python Web框架，专门用于构建RESTful API服务。FastAPI基于Starlette框架构建，兼具Starlette的灵活性和Pydantic的数据验证能力，其核心特点包括：基于Python类型注解的自动数据验证、异步支持带来的高性能、自动生成交互式API文档（基于Swagger UI和ReDoc）、以及完善的OpenAPI支持。

FastAPI的工作原理是通过Python的类型注解（Type Hints）自动推断请求参数和响应模型的数据类型，并利用Pydantic进行数据验证。当用户定义API端点时，只需要使用装饰器（如`@app.get()`、`@app.post()`）标注函数，框架会自动处理路由分发、请求解析、参数验证、响应序列化等繁琐工作。FastAPI还支持依赖注入（Dependency Injection）机制，可以方便地管理数据库连接、认证验证等共享资源。

在性能方面，FastAPI默认运行在ASGI（Asynchronous Server Gateway Interface）服务器上，如Uvicorn或Hypercorn，支持异步请求处理。异步编程允许在等待I/O操作（如数据库查询、网络请求）时释放控制权去处理其他请求，从而显著提高系统的并发处理能力。根据TechEmpower的基准测试，FastAPI在JSON序列化等典型Web场景下的性能表现位居Python框架前列。

### 2.2 psutil 系统监控库

psutil（Process and System Utilities）是一个跨平台的Python库，用于获取系统和进程信息。它能够采集CPU、内存、磁盘、网络、传感器等方面的系统数据，兼容Linux、Windows、macOS等主流操作系统。psutil库提供了丰富的API接口，使用简单方便，是Python实现系统监控的首选工具。

在CPU监控方面，psutil提供了`cpu_percent()`方法用于获取CPU总体使用率，`cpu_percent(percpu=True)`方法则返回各CPU核心的使用率列表。`cpu_times()`方法可以获取CPU的各状态时间（用户态、系统态、空闲态等），通过计算时间差可以得出精确的CPU使用率。`cpu_stats()`方法提供上下文切换、中断等统计信息。

内存监控方面，psutil的`virtual_memory()`方法返回一个包含内存详情的命名元组，包括total（总内存）、available（可用内存）、used（已用内存）、percent（使用率）、buffers（缓冲区）、cached（缓存）等字段。`swap_memory()`方法则提供交换内存的使用情况。对于Linux系统，还可以使用`linux_sys_memory_info()`获取更详细的内存信息，如活动内存、非活动内存、可回收内存等。

磁盘I/O监控通过`disk_io_counters()`和`disk_usage()`两个方法实现。前者返回磁盘读写统计，包括read_count（读操作次数）、write_count（写操作次数）、read_bytes（读字节数）、write_bytes（写字节数）等；后者返回指定路径的磁盘使用情况，包括total、used、free、percent等字段。

网络监控方面，`net_io_counters()`方法返回网络接口的收发统计，包括bytes_sent、bytes_recv、packets_sent、packets_recv、errin、errout等。`net_connections()`方法可以获取当前的网络连接列表，配合`psutil.Process.net_connections()`可以查看特定进程的网络连接情况。

### 2.3 SQLite 数据库

SQLite是一款轻量级的关系型数据库管理系统，采用单一文件存储、无需独立服务器进程的设计，非常适合嵌入式应用和中小型项目。SQLite将整个数据库存储在一个磁盘文件中，支持标准的SQL语法，具备事务ACID特性，性能优异且配置简单。

SQLite的数据库文件格式是跨平台的，在Linux、Windows、macOS等系统间可以无缝迁移。SQLite使用动态类型系统，可以存储任意类型的数据（文本、整数、浮点数、BLOB等），这与Python的数据类型具有良好的兼容性。Python的标准库中自带了sqlite3模块，无需额外安装即可使用。

在监控系统中使用SQLite的主要优势在于：部署简单，无需安装和配置数据库服务器；性能足够，对于中等规模的监控数据存储场景完全能够胜任；单文件存储便于备份和迁移；支持完整的SQL查询能力，便于实现复杂的数据统计和分析需求。

然而，SQLite也存在一些局限性。由于采用文件锁机制，在高并发写入场景下可能成为性能瓶颈；不支持网络远程访问，不适合分布式部署场景。对于本监控系统而言，由于采集频率相对较低（默认每秒一次），且单个监控节点的并发访问量有限，SQLite是理想的选择。

### 2.4 Vue 3 前端框架

Vue.js是一款渐进式JavaScript框架，由尤雨溪于2014年创建。Vue 3是Vue.js的最新 major 版本，在2020年发布，带来了全新的Composition API、性能优化和TypeScript支持。Vue 3的核心设计理念是响应式数据绑定和组件化开发，通过简洁的API帮助开发者构建用户界面。

Composition API是Vue 3引入的重要特性，它提供了一种更加灵活和可复用的代码组织方式。传统的Options API将组件逻辑分散在data、methods、computed、watch等选项中，而Composition API允许开发者使用`setup()`函数集中定义响应式状态、计算属性、方法等，并通过`ref`和`reactive`函数创建响应式数据。Composition API还支持自定义组合函数（Composables），可以将相关逻辑封装为可复用的功能单元。

Vue 3还引入了Teleport（传送门）、Fragments、Suspense等新特性，并大幅优化了虚拟DOM的性能。新版本的响应式系统基于ES6 Proxy实现，相比Vue 2的Object.defineProperty方案，在对象属性添加删除、数组索引访问等方面具有更好的支持。

Vue Router是Vue.js的官方路由管理器，用于实现单页应用（SPA）的页面导航。通过定义路由规则，将URL路径映射到对应的组件，实现无刷新切换页面。Vue Router支持动态路由、嵌套路由、路由守卫等功能，满足复杂应用场景的需求。

Axios是前端常用的HTTP客户端库，用于与后端API进行通信。Axios基于Promise封装，支持请求拦截器、响应拦截器、自动转换JSON数据等功能，并具有良好的错误处理机制。

### 2.5 ECharts 可视化库

ECharts是百度开源的一个基于JavaScript的图表库，能够流畅地在PC和移动设备上运行，兼容当前主流浏览器。ECharts提供了丰富的图表类型，包括折线图、柱状图、饼图、散点图、K线图、地图、热力图、关系图等，能够满足各种数据可视化需求。

ECharts的核心特点是配置式开发，开发者通过设置图表的option对象来定义图表的样式和数据。option对象包含多个配置项，如title（标题）、legend（图例）、tooltip（提示框）、xAxis（X轴）、yAxis（Y轴）、series（数据系列）等。通过修改这些配置项，可以实现图表的个性化定制。

ECharts支持数据动态更新，可以通过`setOption`方法动态修改数据系列，实现图表的实时刷新效果。在监控系统中，可以将采集到的实时数据不断推送到图表中，形成动态变化的效果。ECharts还支持数据的缩放和平移，用户可以拖动图表查看特定时间范围的数据变化。

ECharts的性能经过优化，能够处理大量数据点而不明显影响渲染速度。对于需要展示大量历史数据的场景，可以使用数据采样（sampling）和渐进渲染（progressive rendering）技术来优化性能。ECharts还支持WebGL渲染，可以处理数百万级的数据点，适合大数据可视化场景。

### 2.6 WebSocket 通信协议

WebSocket是一种在单个TCP连接上提供全双工通信的协议，最初在HTML5规范中提出。与传统的HTTP请求-响应模式不同，WebSocket允许服务器主动向客户端推送数据，特别适合实时性要求较高的应用场景。

WebSocket的建立过程始于HTTP握手。客户端发送一个包含Upgrade头的HTTP请求，请求将连接升级为WebSocket协议。服务器如果支持WebSocket，则返回101状态码表示协议切换成功，随后双方可以在这条TCP连接上进行双向通信。WebSocket帧结构紧凑，支持文本帧和二进制帧，最小只需要2字节的开销。

在监控系统中，后端采集到监控数据后，需要立即推送到前端界面进行展示。如果采用轮询方式，前端每隔固定时间间隔向后端发送HTTP请求，不仅实时性差，还会产生大量的HTTP开销。使用WebSocket后，后端可以维持与前端的持久连接，一旦采集到新数据就立即通过连接发送出去，前端收到数据后更新界面，实现秒级甚至毫秒级的数据同步。

FastAPI框架原生支持WebSocket，通过`websocket`路由装饰器可以轻松创建WebSocket端点。Vue 3中可以使用原生的WebSocket API或使用第三方库（如vueuse的useWebSocket）来建立WebSocket连接并进行数据收发。

## 第三章 系统需求分析

### 3.1 功能需求分析

本Linux监控系统的核心目标是为运维人员提供一个便捷的工具，用于实时监控服务器的运行状态。根据项目概述中提到的功能要求，系统需要实现以下具体功能：

**实时数据采集功能**：系统需要按照固定的时间间隔（默认1秒）自动采集服务器的各类监控指标数据，包括CPU使用率、内存使用情况、磁盘I/O统计、网络带宽使用等。采集间隔应该可配置，用户可以根据实际需求调整采集频率。

**实时展示功能**：前端界面需要实时显示各项监控指标的当前值，并通过图表形式展示数据的变化趋势。CPU监控需要支持查看各核心的独立使用率；内存监控需要显示总量、已用、可用和百分比；磁盘I/O需要显示读写速率；网络需要显示上行和下行带宽。

**历史数据查询功能**：用户可以查询特定时间段内的监控数据，系统从数据库中检索相应记录并以图表形式展示。查询功能应该支持时间范围选择，包括最近一小时、最近一天、最近一周、自定义时间范围等选项。

**告警功能**：用户可以为各项监控指标设置告警阈值。当实时采集的数据超过用户设定的阈值时，系统应该产生告警记录，并在前端界面上显示告警提示。告警信息应该包含告警时间、指标名称、当前值、阈值等信息。用户可以查看历史告警记录。

## 第四章 系统总体设计

### 4.1 系统架构设计

本系统采用前后端分离的B/S（Browser/Server）架构，整体架构分为三层：数据采集层、业务逻辑层和表示层。

数据采集层位于服务器端，负责从操作系统获取各类监控数据。该层的核心组件是数据采集模块，利用psutil库提供的系统API定时采集CPU、内存、磁盘、网络等信息。采集到的数据一部分直接推送到前端展示，另一部分写入数据库持久化存储。

业务逻辑层同样位于服务器端，是系统的核心处理层。该层基于FastAPI框架构建RESTful API和WebSocket端点，提供数据查询、告警判断、数据导出等业务功能。FastAPI作为ASGI应用运行在Uvicorn服务器上，能够高效处理HTTP请求和WebSocket连接。

表示层即前端浏览器端，采用Vue 3单页应用架构。前端通过Axios向后端API发送HTTP请求获取数据，通过WebSocket客户端接收后端推送的实时数据，使用ECharts将数据渲染为可视化图表。

### 4.2 模块划分

系统主要包含以下模块：

一、监控数据采集模块（collector.py）。负责使用 psutil 库采集系统的各项监控指标，包括 CPU 使用率、内存使用情况、磁盘 I/O 统计、网络带宽统计和进程信息。该模块提供 get_cpu()、get_memory()、get_disk()、get_network()、get_processes() 等函数接口。

二、数据库模块（database.py）。负责管理 SQLite 数据库连接，创建数据表结构，提供数据存储和查询接口。数据库模块包含 init_db()、save_metrics()、get_history_metrics() 等函数。

三、API 路由模块（metrics.py、websocket.py、alerts.py、processes.py、export.py）。负责定义 REST API 端点和 WebSocket 端点，处理前端请求，返回相应的监控数据。

四、前端组件模块。包含 Dashboard.vue（实时监控仪表盘）、History.vue（历史数据查询）、Processes.vue（进程监控）、Alerts.vue（告警管理）、Export.vue（数据导出）等页面组件，以及 CpuChart.vue、MemoryGauge.vue 等图表组件。

### 4.3 数据库设计

系统使用 SQLite 数据库存储监控数据，主要包含以下四张数据表：

一、CPU 统计表（cpu_stat）。用于存储 CPU 使用率数据，包含 id（主键）、timestamp（时间戳）、cpu_percent（CPU 使用率）、cpu_per_core（各核心使用率 JSON 字符串）、created_at（创建时间）字段。

二、内存统计表（memory_stat）。用于存储内存使用数据，包含 id（主键）、timestamp（时间戳）、total（总内存）、available（可用内存）、percent（使用率）、used（已用内存）、created_at（创建时间）字段。

三、网络统计表（net_stat）。用于存储网络 I/O 数据，包含 id（主键）、timestamp（时间戳）、bytes_sent（发送字节数）、bytes_recv（接收字节数）、pack_sent（发送包数）、pack_recv（接收包数）、created_at（创建时间）字段。

四、磁盘 I/O 统计表（disk_stat）。用于存储磁盘 I/O 数据，包含 id（主键）、timestamp（时间戳）、read_count（读操作次数）、write_count（写操作次数）、read_bytes（读字节数）、write_bytes（写字节数）、created_at（创建时间）字段。

各表在 timestamp 字段上建立索引，以加快历史数据查询速度。

### 4.4 API 接口设计

系统提供以下 REST API 接口：

/metrics/current：获取当前所有指标，返回 CPU、内存、磁盘、网络的当前值。

/metrics/cpu：获取 CPU 指标，返回总体使用率和各核心使用率。

/metrics/memory：获取内存指标，返回总内存、可用内存、使用率等。

/metrics/disk：获取磁盘 I/O 指标，返回读写统计信息。

/metrics/network：获取网络 I/O 指标，返回网络带宽统计。

/metrics/history：获取历史指标，支持指定指标类型、开始时间、结束时间和返回条数。

/metrics/system/info：获取系统硬件信息，包括 CPU 型号、核心数、内存总量、磁盘总量等。

/processes：获取进程列表，支持按 CPU 或内存排序。

/alerts/rules：获取告警规则列表。

/alerts/rules/{name}/toggle：切换告警规则启用状态。

/config：获取或更新系统配置。

/export/csv：导出 CSV 格式的历史数据。

WebSocket 端点：/ws/metrics，支持 subscribe（订阅）、unsubscribe（取消订阅）、ping/pong（心跳）等消息类型。

#### 5.1.2 内存采集实现

内存采集使用 psutil.virtual_memory() 方法获取内存信息，返回一个命名元组，包含 total（总内存）、available（可用内存）、used（已用内存）、percent（使用率）等字段。代码将字节单位转换为 GB（除以 1024 的三次方），并保留两位小数。

#### 5.1.3 磁盘 I/O 采集实现

磁盘 I/O 采集使用 psutil.disk_io_counters() 方法获取磁盘读写统计，返回 read_count、write_count、read_bytes、write_bytes 等字段。同时计算读写速率，将字节转换为 MB。

#### 5.1.4 网络 I/O 采集实现

网络 I/O 采集使用 psutil.net_io_counters() 方法获取网络接口统计，返回 bytes_sent、bytes_recv、packets_sent、packets_recv 等字段。计算发送和接收速率，将字节转换为 KB。

#### 5.1.5 进程监控实现

进程监控使用 psutil.process_iter() 方法遍历系统进程，获取每个进程的 pid、name、cpu_percent、memory_percent、status 等信息。代码过滤掉 cpu_percent 和 memory_percent 为 None 的进程，按 CPU 使用率降序排序，返回前 N 个进程。

### 5.2 后端服务模块实现

#### 5.2.1 FastAPI 应用主入口

后端主入口 main.py 创建 FastAPI 应用，配置 CORS 中间件，注册各模块路由，并使用 lifespan 上下文管理器管理应用生命周期。应用启动时初始化数据库，启动后台任务；应用关闭时停止后台任务。

#### 5.2.2 REST API 路由实现

API 路由模块定义了各业务功能的端点。metrics.py 包含指标相关的 GET 接口，processes.py 包含进程相关的接口，alerts.py 包含告警管理接口，export.py 包含数据导出接口。各路由使用 FastAPI 的依赖注入机制获取所需数据。

### 5.3 WebSocket 实时推送实现

WebSocket 模块实现实时数据推送功能。ConnectionManager 类管理所有活跃的 WebSocket 连接，维护连接列表和订阅信息。websocket_metrics 端点处理客户端连接，接收订阅消息，根据订阅的指标类型定时推送数据。

客户端可以通过发送 {"action": "subscribe", "metrics": ["cpu", "memory"], "interval": 1} 消息订阅指标，通过发送 {"action": "unsubscribe"} 取消订阅。后端定时采集数据并推送到所有订阅的客户端。

### 5.4 前端可视化模块实现

前端采用 Vue 3 + ECharts 实现可视化展示。项目目录结构包含 components 目录（图表组件）和 views 目录（页面视图）。

Dashboard.vue 是系统的主页面，集成各类图表组件。CpuChart.vue、MemoryGauge.vue、DiskChart.vue、NetworkChart.vue 分别负责 CPU、内存、磁盘、网络的图表展示。图表使用 ECharts 的配置式开发方式，通过设置 option 对象定义图表样式和数据。

前端通过 Axios 发送 HTTP 请求获取历史数据，通过原生 WebSocket API 接收实时推送数据。Composition API 的 setup() 函数集中管理响应式状态和业务逻辑。

## 第七章 部署与使用

### 7.1 环境要求

系统部署需要以下环境：

后端：Python 3.10+，可使用 pyenv 或 conda 管理 Python 版本。

前端：Node.js 18+，可使用 nvm 管理 Node.js 版本。

操作系统：Linux（推荐 Ubuntu 20.04 或 CentOS 8）。

### 7.2 后端部署步骤

第一步，进入后端目录并创建虚拟环境。执行 python3 -m venv .venv 命令创建虚拟环境，然后执行 source .venv/bin/activate 激活虚拟环境。

第二步，安装依赖。执行 pip install -r requirements.txt 安装项目依赖，包括 fastapi、uvicorn、psutil、websockets、pydantic 等。

第三步，启动服务。执行 uvicorn main:app --host 0.0.0.0 --port 8000 启动后端服务。服务启动后可访问 http://localhost:8000/docs 查看 API 文档。

### 7.3 前端部署步骤

第一步，进入前端目录。执行 cd frontend 命令进入前端项目目录。

第二步，安装依赖。执行 npm install 安装项目依赖。

第三步，开发模式。执行 npm run dev 启动开发服务器，访问 http://localhost:5173 查看系统。

第四步，生产构建。执行 npm run build 构建生产版本，生成的静态文件位于 dist 目录。

### 7.4 Nginx 反向代理配置

如需通过域名访问系统，可配置 Nginx 反向代理。配置文件示例：

```nginx
server {
    listen 80;
    server_name monitor.example.com;

    # 前端静态资源
    location / {
        root /var/www/monitor/dist;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # REST API 代理
    location /api/ {
        proxy_pass http://127.0.0.1:8000/;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # WebSocket 代理
    location /ws/ {
        proxy_pass http://127.0.0.1:8000/ws/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "Upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;
    }
}
```
配置详解：在上述 Nginx 配置中，前端静态资源通过 `root` 指令指定路径，并使用 `try_files` 支持 Vue Router 的 History 模式。API 请求代理至后端的 8000 端口，并传递客户端的真实 IP。WebSocket 代理则特别配置了 `Upgrade` 和 `Connection` 头部，以维持长连接的稳定性，同时设置了较长的超时时间防止连接意外断开。

## 第八章 总结与展望

### 8.1 总结

本文设计并实现了一套基于 FastAPI + Vue3 的 Linux 运行监控系统，主要完成了以下工作：

一、实现了 CPU、内存、磁盘 I/O、网络带宽的实时监控采集。使用 psutil 库封装了数据采集模块，能够准确获取系统各项指标。

二、使用 SQLite3 实现了历史数据的存储和查询。设计了合理的数据库表结构，建立了时间戳索引，支持高效的历史数据查询。

三、基于 FastAPI 构建了 REST API 和 WebSocket 服务。提供了完整的 API 接口，支持 HTTP 请求和 WebSocket 实时推送。

四、使用 Vue3 和 ECharts 实现了可视化仪表盘。设计了友好的用户界面，实时展示各项监控指标。

五、实现了告警机制、进程监控和数据导出功能。完善了系统的功能体系，提升了实用性。

系统经过测试，各项功能满足设计要求，运行稳定。

### 8.2 不足与改进

一、扩展性不足。目前仅支持单节点监控，可考虑分布式架构支持多服务器监控。

二、告警通知方式单一。当前仅支持界面提示，可增加邮件、短信等通知方式。

三、数据可视化功能可增强。可增加更多图表类型和自定义仪表盘功能。

四、性能可进一步优化。可引入 Redis 缓存和数据压缩技术。

### 8.3 展望

随着云计算和 DevOps 理念的深入，监控系统将在以下方向继续发展：

一、容器化监控。支持 Docker、Kubernetes 监控，适应云原生环境。

二、智能化告警。基于机器学习的异常检测，实现更精准的告警。

三、云原生架构。微服务化和容器化部署，提升系统的可扩展性和可靠性。
